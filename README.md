# Provenance Guard

A backend system any creative-sharing platform could plug in to answer one
question about a piece of text: **was this written by a human, or generated by
AI?** It classifies submitted text with two independent signals, scores its
confidence honestly (it is allowed to say "I'm not sure"), shows a plain-language
transparency label, lets creators appeal, and records every decision in a
structured audit log.

The guiding principle, taken from the project hints: **on a writing platform, a
false positive — accusing a human of using AI — is the worst outcome.** That one
idea shapes the high AI threshold, the wide "uncertain" band, the conservative
LLM prompt, and the appeal path.

## Architecture

The full diagram (submission flow and appeal flow) lives in
[planning.md](planning.md) under the **Architecture** section. In short:

A submission enters through `POST /submit` (Flask), passes the **rate limiter**,
then runs through **two detection signals** — an LLM classifier (Groq) and a
pure-Python stylometric analyzer. The **confidence scorer** blends their two
scores into one number, the **label maker** turns that number into plain-language
text, the decision is written to the **audit log**, and the response returns the
result, score, and label. In the appeal flow, `POST /appeal` flips the content's
status to `under_review`, logs the appeal next to the original decision, and
returns a confirmation — no automatic re-scoring happens.

**Components / files:**

| File | Role |
|------|------|
| [app.py](app.py) | Flask app, routes, rate limiting |
| [signals.py](signals.py) | Signal 1 (Groq LLM) and Signal 2 (stylometry) |
| [scoring.py](scoring.py) | Combines the two signals into one confidence score + band |
| [labels.py](labels.py) | Maps score/band to the three transparency-label variants |
| [audit_log.py](audit_log.py) | Append-only JSONL audit log + mutable content-status store |
| [planning.md](planning.md) | Design doc, architecture diagram, AI tool plan |

## Setup and running

Requirements: Python 3.11+, a free [Groq](https://console.groq.com) API key.

```bash
# 1. Install dependencies (a virtualenv is recommended)
pip install -r requirements.txt

# 2. Add your Groq key to a .env file in the repo root
echo "GROQ_API_KEY=your_key_here" > .env

# 3. Run the server
python app.py            # serves on http://127.0.0.1:5000
```

## Detection signals (and why these two)

The system uses **two signals that measure genuinely different things**. One is
semantic (meaning), one is structural (statistics). Because they are
independent, the combination is more informative than either alone — when one is
fooled, the other often is not.

### Signal 1 — LLM classification (Groq `llama-3.3-70b-versatile`)

- **What it measures:** whether the text *reads* like human or AI writing, taken
  as a whole — tone, rhythm, specificity, personal voice. Output: a 0–1 score
  (higher = more likely AI).
- **Why I chose it:** the strongest clue for AI text is holistic and semantic
  ("this sounds generic and over-hedged"), which a capable LLM can sense and a
  word-counting heuristic cannot.
- **Blind spot:** it conflates *polished/formal* writing with AI, and can be
  fooled by lightly edited AI text. It is also slightly non-deterministic.

### Signal 2 — Stylometric heuristics (pure Python)

- **What it measures:** three countable properties —
  - **sentence-length burstiness** (coefficient of variation of sentence word
    counts) — the strongest stylometric tell,
  - **type-token ratio** (vocabulary diversity),
  - **punctuation density**.

  Each is mapped to a 0–1 "AI-likeness" and combined (burstiness 60%, TTR 25%,
  punctuation 15%). Output: a 0–1 score.
- **Why I chose it:** it is fully independent of the LLM — it doesn't read
  meaning at all, just structure. AI text tends to be *uniform* (even sentence
  lengths, regular punctuation); human writing is bumpier. It is also free,
  instant, and deterministic.
- **Blind spot:** it is shallow and noisy on short text, and it flags plain or
  repetitive human writing as AI because that writing is also statistically
  uniform.

**Why two and not one:** single-signal AI detection is brittle. Pairing a
semantic signal with a structural one means a text has to look AI-generated *in
two different ways* before the score climbs — which is exactly what protects
human writers.

**What I'd change for real deployment:** the LLM signal is the bottleneck (cost,
latency, non-determinism, and a known false-positive tendency on formal prose).
For production I'd add a third, trained signal (e.g. a fine-tuned classifier or
a perplexity-based detector), calibrate scores against a labeled dataset rather
than hand-tuned constants, and treat the LLM as one voice in an ensemble rather
than the dominant one.

## Confidence scoring (and why this approach)

Both signals output 0–1 (higher = more likely AI). They are blended:

```
final_score = (0.70 * llm_score) + (0.30 * stylometric_score)
```

For very short text (< 40 words) the weights become **0.80 / 0.20**, because the
stylometric signal is unreliable on little text.

The score maps to **three bands**:

| Final score | Band | Meaning |
|-------------|------|---------|
| `>= 0.70` | `likely_ai` | high confidence it is AI |
| `0.40 – 0.69` | `uncertain` | not sure either way |
| `< 0.40` | `likely_human` | high confidence a human wrote it |

**Why this approach.** The confidence number is a *design decision before a
technical one*. I decided what I wanted the score to mean first: a number near
0.5 should produce a visibly different, hedged label than a number near 0.95.
The high `0.70` AI threshold is deliberate — it means a borderline text becomes
"uncertain," not "AI." This is where the false-positive asymmetry lives.

**Why these weights.** My original plan was 0.6 / 0.4. During Milestone 4
calibration I found the stylometric signal was noisy and scored clearly-AI text
only ~0.48, which dragged genuine AI down to "uncertain." The data said the LLM
is the more reliable signal, so I shifted to 0.7 / 0.3. The high threshold —
not a low LLM weight — is what guards against false positives. (planning.md was
updated to match.)

### Two example submissions (real scores from testing)

**High-confidence case — clearly human, casual text:**

> "ok so i finally tried that new ramen place downtown and honestly?
> underwhelming. the broth was fine but they put WAY too much sodium in it..."

| llm_score | stylometric_score | **final_score** | band |
|-----------|-------------------|-----------------|------|
| 0.20 | 0.18 | **0.193** | likely_human |

**Lower-confidence case — formal human writing about monetary policy:**

> "The relationship between monetary policy and asset price inflation has been
> extensively studied in the literature. Central banks face a fundamental
> tension between their mandate for price stability..."

| llm_score | stylometric_score | **final_score** | band |
|-----------|-------------------|-----------------|------|
| 0.60 | 0.59 | **0.598** | uncertain |

The score moves from **0.193** to **0.598** across these two human-written
inputs — it is not a constant. The casual piece is confidently human; the formal
piece lands in "uncertain" precisely because formal prose is the hard case (see
[Known limitations](#known-limitations)). For contrast, a clearly AI-generated
paragraph scored **0.705 (likely_ai)**.

**How I tested that scores are meaningful.** I ran four deliberately chosen
inputs — clearly AI, clearly human, formal human, and lightly-edited AI —
printing each signal *separately* so I could see which one drove each result.
This caught a real bug (the LLM flagging formal human writing as AI) and let me
confirm the final scores span all three bands with a wide gap, rather than
clustering near 0.5.

## Transparency label variants

The label shown to a reader **changes with the confidence score**. `{score}` is
the final score shown as a percent. All three variants are written out verbatim
below.

**High-confidence AI** (`final_score >= 0.70`):

> 🤖 Likely AI-generated. Our checks strongly suggest this text was created with AI tools (confidence: {score}%). This is an automated estimate, not proof. If you believe this is wrong, you can appeal.

**High-confidence human** (`final_score < 0.40`):

> ✍️ Likely human-written. Our checks suggest a person wrote this text (confidence it is AI: {score}%). This is an automated estimate, not a guarantee.

**Uncertain** (`final_score` 0.40–0.69):

> ❓ Uncertain. Our checks could not confidently tell whether a human or AI wrote this text (confidence it is AI: {score}%). We are showing this honestly rather than guessing. The creator may add context or appeal.

**Design notes.** Every label says it is an *automated estimate, not proof* —
honesty is built into the wording. The AI label is the only one that actively
invites an appeal, because that is the label that can hurt a creator. The
"uncertain" label explicitly says we are *showing this honestly rather than
guessing*, so a reader understands the system is admitting doubt rather than
hiding it.

## Appeals workflow

- **Who can appeal:** the creator (referenced by `content_id`).
- **What they provide:** the `content_id` and a written `creator_reasoning`.
- **What the system does:** confirms the content exists, sets its status to
  `under_review` in the content store, writes the appeal to the audit log next to
  the original decision, and returns a confirmation. **No automated
  re-classification** — a human decides later. A reviewer can open `GET /queue`
  to see every under-review item with its full context (original text, both
  signal scores, the label shown, and the creator's reasoning).

Test it with a `content_id` from any `/submit` response:

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-CONTENT-ID", "creator_reasoning": "I wrote this myself from personal experience. I am a non-native English speaker and my writing style may appear more formal than typical."}'
```

Response:

```json
{
  "content_id": "dc2f291e-8a92-425d-8599-0150fe57d292",
  "status": "under_review",
  "appealed": true,
  "message": "Appeal received. This content is now under review by a human moderator. No automated re-classification is performed.",
  "original_attribution": "uncertain",
  "original_confidence": 0.5978
}
```

## Rate limiting

Implemented with **Flask-Limiter** (in-memory storage) on `POST /submit`,
keyed per client IP:

```
@limiter.limit("10 per minute;100 per day")
```

**Chosen limits and reasoning (not arbitrary):**

- **10 per minute** — a real writer submits their own work occasionally, not in
  bursts. Ten per minute is generous for legitimate use (re-checking after an
  edit, trying a couple of drafts) while stopping a script that hammers the
  endpoint.
- **100 per day** — caps sustained automated abuse while still allowing a heavy
  but legitimate day (e.g. a platform spot-checking a backlog).
- **Cost angle:** every `/submit` triggers a Groq API call, so the limit also
  protects the upstream free-tier quota from being drained by a flood.
- **For real deployment** I'd key the limit on the authenticated account, not
  the IP (IPs are shared behind NAT and easy to rotate), and move storage from
  `memory://` to Redis so limits survive restarts and span multiple workers.

**Evidence** — sending 12 rapid requests (limit is 10/min) produced exactly:

```

request 2  -> 200
request 3  -> 200
request 4  -> 200
request 5  -> 200
request 6  -> 200
request 7  -> 200
request 8  -> 200
request 9  -> 200
request 10 -> 200
request 11 -> 429
request 12 -> 429
```

recent entries. Each classification records the timestamp, content id,
attribution, confidence, **both** individual signal scores, the raw stylometric
metrics, the weights used, the label, and whether it has been appealed. Appeals
are logged as their own event linked by `content_id`.

## Known limitations

AI detection is an unsolved problem; this system acknowledges uncertainty rather
than pretending to be perfect. Specific cases it will likely get wrong:

1. **Formal writing by a careful or non-native-English human.** This is the
   sharpest one, and it is tied directly to both signals' properties. The
   stylometric signal reads *low sentence-length burstiness* as AI — and formal
   prose is uniform by nature (in testing, the monetary-policy paragraph had the
   *lowest* burstiness of all samples, `cv=0.256`). At the same time the LLM
   signal associates a polished, even tone with AI. So both signals push the
   same wrong direction. The mitigation is the high 0.70 threshold (it lands in
   "uncertain," not "AI") plus the appeal path — but it will never confidently
   call this kind of text human.

2. **Minimalist or heavily repetitive poetry.** Short poems use few words,
   repeat lines, and keep line lengths even. The stylometric signal sees low
   vocabulary diversity (low type-token ratio) and low burstiness — the same
   fingerprint as AI — and on short text it carries more noise. A genuine human
   poem can therefore be pushed upward.

3. **Lightly edited AI text.** A person who rewrites a few sentences of AI output
   breaks both signals at once: the text gains some human bumpiness but keeps the
   AI smoothness. It usually lands in "uncertain," which is the honest answer,
   but it will rarely be caught confidently.

## Spec reflection

**One way the spec guided the implementation.** The hint that *a false positive
is worse than a false negative on a writing platform* drove concrete design
choices: the AI threshold is set high (0.70) rather than at the midpoint, the
"uncertain" band is wide, the LLM prompt is explicitly told that formality is
not proof of AI, and only the AI label invites an appeal. Without that single
sentence in the spec I would probably have used a naive 0.5 cutoff and shipped a
system that punishes good human writers.

**One way the implementation diverged from the spec.** The spec's audit-log
example shows a single record whose `status` is updated in place. I instead kept
the audit log strictly **append-only** (an immutable history of events) and put
the mutable `status` in a *separate* content store (`contents.json`) that the
appeal endpoint updates. I diverged because an audit log you can edit isn't
really an audit log — for a trust-and-attribution system, the record of what was
decided and when should never be rewritten, even when status changes. The
mutable state and the immutable history are two different concerns, so I stored
them separately.

## AI usage

I used an AI coding assistant (Claude Code) throughout, but reviewed and
corrected its output against my planning doc rather than pasting it blindly. Two
specific instances:

1. **LLM detection prompt — overrode a false-positive bug.** I directed the AI
   to generate the Groq classification prompt and signal function. Its first
   version produced reasonable code, but when I tested it the prompt scored a
   *formal human* paragraph at **0.80 (AI)** — exactly the false positive the
   spec warns against. I overrode the prompt, rewriting the system message to
   state that formal/polished writing is not evidence of AI and to look for
   specific AI tells (filler phrases, hedging, lack of concrete detail). After
   the change that same text dropped to **0.60 (uncertain)** while clearly-AI
   and clearly-human cases were unaffected.

2. **Scoring weights — overrode the AI's faithful-but-wrong implementation.** I
   asked the AI to implement the confidence scorer using the 0.6 / 0.4 weights
   from my planning.md. It did so correctly — but calibration testing showed
   clearly-AI text only reached **0.673**, falling into "uncertain." The AI had
   implemented my spec faithfully; my spec was the problem. I overrode the
   weights to **0.7 / 0.3** based on the real numbers (the stylometric signal
   was noisy and scored genuine AI low), verified all three bands were reachable,
   and updated planning.md so the doc and code stayed in sync.

(Additionally, the AI generated the stylometric function, but I calibrated its
threshold constants myself by first running an exploration script to see the
real metric values on sample texts, rather than accepting guessed numbers.)
